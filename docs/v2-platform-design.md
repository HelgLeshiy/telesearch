# telesearch v2 — технический дизайн платформы и план реализации

> Статус: предложение (design proposal). Документ описывает целевую
> архитектуру превращения `telesearch` из локальной однопользовательской
> CLI‑утилиты в многопользовательский сервис, наилучшую техническую
> реализацию каждого из пяти направлений и поэтапный план внедрения.

## 0. Контекст и принципы

### 0.1. Что есть сейчас (v1)

`telesearch` сегодня — это **движок**, а не сервис:

- Однопользовательский, одно‑экспортный, локальный пайплайн. Один процесс,
  один синглтон `Settings` (`telesearch/config.py`), одна таблица LanceDB
  `chunks` (`telesearch/index/store.py`), одно embedding‑пространство.
- Источник‑агностичное ядро: всё ниже `Message`/`Chunk`
  (`telesearch/models.py`) не знает, откуда пришли данные. Парсер Telegram
  (`telesearch/ingest/telegram_parser.py`) — единственная реализация,
  абстракции над ним нет, но шов для неё уже фактически существует.
- Поиск сильный: hybrid RRF (dense bge‑m3 + BM25) → cross‑encoder rerank
  (`telesearch/search/retriever.py`), плюс HyDE и добор соседей в RAG
  (`telesearch/search/rag.py`).
- Тяжёлая часть — индексация мультимодалки (Qwen2.5‑VL + Whisper), требует GPU.
  Поиск/rerank лёгкие, идут и на CPU.
- Нет ничего «сервисного»: ни HTTP API, ни пользователей, ни сессий, ни
  фоновых задач, ни изоляции. Streamlit (`telesearch/ui/app.py`) — read‑only
  обёртка над `Retriever`.

### 0.2. Ключевой вывод

Пять идей пользователя — это «верхушка». Под ними нужно построить «низ»,
которого нет вообще: API‑слой, реляционная БД, очередь задач, объектное
хранилище, изоляция данных, наблюдаемость. Поэтому дизайн разделён на:

- **Сквозные фундаменты** (раздел 2) — то, на чём держатся все пять пунктов.
- **Реализация по пунктам** (разделы 3–7) — наилучшая техническая реализация
  каждого направления, поверх фундаментов.
- **План по фазам** (раздел 9) — порядок внедрения с критериями готовности.

### 0.3. Инженерные принципы

1. **Сохранить ядро, добавить обёртку.** Бизнес‑логика индексации/поиска не
   переписывается, а извлекается из CLI в переиспользуемый сервисный слой.
2. **Изоляция данных важнее любой фичи.** Хранятся чужие личные переписки.
   Цена ошибки доступа — катастрофа, а не баг.
3. **Всё тяжёлое — асинхронно.** Никакой синхронной индексации в HTTP‑запросе.
4. **Каждый внешний слой — за интерфейсом** (векторный стор, парсеры,
   LLM/VLM/ASR), чтобы можно было менять реализацию без переписывания.
5. **Дешёвое по умолчанию, дорогое — по запросу.** Текстовая индексация (CPU)
   доступна всем; мультимодалка (GPU) — опционально/в платном тарифе.

### 1.1. Технологический стек и обоснование

| Слой | Выбор | Почему |
|---|---|---|
| API | **FastAPI** | Уже есть `pydantic`/`openai` в зависимостях; async, авто‑OpenAPI, WebSocket для прогресса. |
| Реляционная БД | **PostgreSQL** + SQLAlchemy 2.x + Alembic | Транзакции, JSONB, расширение `pgvector` про запас, зрелость. |
| Очередь | **Redis + Arq** (или Celery) | Arq async‑native, лёгкий; Celery — если нужны зрелые фичи (rate‑limit, маршрутизация очередей). Выбор: Arq для простоты, Celery если масштаб потребует. |
| Объектное хранилище | **S3‑совместимое** (MinIO локально, S3/R2 в проде) | Аплоады и медиа не должны лежать в БД; presigned URL для прямой загрузки. |
| Векторный стор | **LanceDB** (изоляция per‑workspace), интерфейс под замену на Qdrant/pgvector | Сохраняем текущий движок; интерфейс снимает риск масштаба. |
| Аутентификация | **OIDC** (Authentik/Keycloak self‑host или Auth0) + JWT сессии | Не писать крипто/пароли руками. |
| Фронтенд | **React + TypeScript + Vite**; граф — **Cosmograph/Sigma.js/deck.gl** | WebGL‑рендер графа на десятки тысяч узлов; Streamlit для прода непригоден. |
| Наблюдаемость | **OpenTelemetry + Prometheus + structured logging** | Многопроцессность требует трассировки задач. |

> Streamlit (`telesearch/ui/app.py`) остаётся как внутренний admin/debug‑UI,
> но не как продуктовый фронтенд.

## 2. Сквозные фундаменты

Это «нулевые» работы, разблокирующие все пять пунктов. Делать в первую очередь
(см. Фаза 0/1), потому что они почти безрисковы и меняют форму кода под всё
остальное.

### 2.1. Абстракция источников: `Source` / `Parser`

**Проблема.** Сейчас `parse_export` жёстко завязан на Telegram. Пункт 3 (любые
загрузки + парсеры мессенджеров) невозможен без абстракции.

**Решение.** Ввести протокол парсера, который любую загрузку превращает в поток
`Message` (модель уже источник‑агностична). Реестр парсеров + автодетект
формата.

```python
# telesearch/ingest/base.py
from __future__ import annotations
from dataclasses import dataclass
from pathlib import Path
from typing import Iterator, Protocol

from ..models import Message


@dataclass
class SourceContext:
    """Где лежат загруженные данные и метаданные о загрузке."""
    root: Path                 # распакованный корень загрузки
    source_id: str             # id записи Source в Postgres
    workspace_id: str
    declared_kind: str | None  # подсказка пользователя: "telegram"|"whatsapp"|...


class Parser(Protocol):
    name: str                  # "telegram", "whatsapp", "vk", "generic_text", ...

    def sniff(self, ctx: SourceContext) -> float:
        """Вернуть уверенность 0..1, что эта загрузка — наш формат."""

    def parse(self, ctx: SourceContext) -> Iterator[Message]:
        """Привести загрузку к потоку нормализованных Message."""
```

Реестр и выбор парсера:

```python
# telesearch/ingest/registry.py
_PARSERS: list[Parser] = []

def register(parser: Parser) -> None:
    _PARSERS.append(parser)

def select_parser(ctx: SourceContext) -> Parser:
    if ctx.declared_kind:
        for p in _PARSERS:
            if p.name == ctx.declared_kind:
                return p
    # автодетект по максимальной уверенности sniff()
    ranked = sorted(_PARSERS, key=lambda p: p.sniff(ctx), reverse=True)
    return ranked[0]  # generic_* всегда возвращает >0 → дефолтный fallback
```

Существующий код переезжает в `TelegramParser(Parser)`, оборачивая текущий
`parse_export`. Это первый и самый дешёвый рефактор.

Дополнительно: расширить `Message` парой полей, чтобы поддержать не‑Telegram
источники без ломки схемы:

```python
# дополнения к Message (telesearch/models.py)
source_kind: str = "telegram"      # откуда пришло
external_id: str | None = None     # исходный id в системе-источнике
thread: str | None = None          # тред/канал внутри источника
```

### 2.2. Векторный стор за интерфейсом + мульти‑тенантность

**Проблема.** `VectorStore` хардкодит одну таблицу `chunks` в одном
`settings.db_path`. Для многих пользователей нужна изоляция и параметризация.

**Решение — двухуровневая модель хранения:**

- **БД на воркспейс** (физический каталог LanceDB): `data/ws/<workspace_id>/lancedb`.
  Жёсткая физическая изоляция: чужие данные невозможно прочитать, даже если
  забыть фильтр. Легко удалить/экспортировать данные пользователя (важно для
  приватности и GDPR‑права на удаление).
- **Коллекции внутри воркспейса** — поле `collection_id` в схеме (источник =
  одна коллекция). Поиск можно скоупить на подмножество коллекций.

Почему не «один общий индекс + фильтр `owner_id`»: изоляция держалась бы лишь
на корректности `where`, а цена ошибки — чужая личная переписка. Физическая
изоляция убирает целый класс утечек. Кросс‑воркспейс поиск (шеринг, пункт 4)
решается на уровне приложения через multi‑store fan‑out (раздел 6.3).

Интерфейс:

```python
# telesearch/index/base.py
class VectorBackend(Protocol):
    def add(self, rows: list[dict], vectors: np.ndarray) -> None: ...
    def build_fts(self) -> None: ...
    def hybrid_search(self, query_text, query_vec, *, k, where: str | None) -> list[dict]: ...
    def fetch_around(self, ...) -> list[dict]: ...
    def delete(self, where: str) -> int: ...
    def drop(self) -> None: ...
```

`LanceVectorStore(VectorBackend)` — текущая реализация, параметризованная путём
воркспейса. Позже можно добавить `QdrantBackend`/`PgVectorBackend` без правок
вызывающего кода.

**Расширение схемы** (`telesearch/index/store.py`, `_schema`) — добавить поля
для мульти‑тенантности, источников и фильтрации:

```python
pa.field("collection_id", pa.string()),  # = source_id; скоуп поиска
pa.field("source_kind", pa.string()),    # telegram|whatsapp|file|...
pa.field("doc_id", pa.string()),         # группировка чанков одного файла/чата
pa.field("lang", pa.string()),           # язык (для фильтров/моделей)
# timestamp/modality/sender уже есть → используются как фильтры
```

> Важно: попутно чинится известный баг из `AGENTS.md` —
> `if TABLE_NAME in self.db.list_tables().tables` ненадёжен на новых версиях
> LanceDB; заменить на `self.db.table_names()` (плоский `list[str]`).

### 2.3. Конфигурация: от синглтона к контексту

`get_settings()` — процессный синглтон, чего достаточно для глобальных
настроек (модели, устройства), но **не** для per‑request/per‑workspace
параметров. Ввести `RequestContext` (workspace, user, права, выбранные
коллекции), который пробрасывается в `Retriever`/индексацию. Глобальный
`Settings` остаётся для модельной инфраструктуры.

### 2.4. Извлечение сервисного слоя из CLI

Логику из `cli.index` / `cli.search` / `cli.ask` вынести в
`telesearch/service/` (чистые функции/классы), которые одинаково вызываются из:
CLI (обратная совместимость), API‑эндпоинтов и воркеров очереди. CLI становится
тонкой обёрткой — как сейчас, но над сервисом, а не над `build_index` напрямую.

## 3. Пункт 1 — Авторизация, пользователи, шеринг, изоляция данных

**Вероятность технического успеха: ~99% (построить) / ~95% (хорошо).** Решённая
инженерная задача; основной риск — приватность, не код.

### 3.1. Доменная модель (PostgreSQL)

```
users(id, email, name, oidc_sub, created_at, status)
workspaces(id, owner_user_id, name, plan, created_at)
memberships(id, workspace_id, user_id, role)         -- owner|admin|member|viewer
sources(id, workspace_id, kind, name, status,        -- загруженный источник
        bytes, created_by, created_at)                --   = коллекция в векторном сторе
documents(id, source_id, external_id, kind, meta_jsonb)
jobs(id, workspace_id, source_id, type, state,        -- очередь индексации
     progress, error, created_at, updated_at)
shares(id, resource_type, resource_id,                -- шеринг
       principal_type, principal_id, role, created_at)
graph_snapshots(id, workspace_id, params_hash,        -- кэш графа (пункт 5)
                state, storage_key, created_at)
audit_log(id, workspace_id, user_id, action,          -- доступ к чувствительным данным
          resource, ts, ip)
```

Иерархия доступа: **user → membership → workspace → source(=collection) →
documents**. Векторные данные физически разложены по `workspace_id` (раздел
2.2).

### 3.2. Аутентификация и сессии

- **OIDC** (внешний или self‑host IdP). Не хранить пароли. На входе —
  обмен кода на токены, создание/линк `users.oidc_sub`.
- API принимает короткоживущий **JWT access** + refresh; на сервере —
  валидация подписи + проверка членства в воркспейсе при каждом запросе.
- Middleware FastAPI собирает `RequestContext` (user, активный workspace,
  роль, разрешённые коллекции) до попадания в бизнес‑логику.

### 3.3. RBAC и проверка доступа

Декоратор/зависимость уровня эндпоинта:

```python
async def require_workspace(role: Role):
    async def dep(ctx: RequestContext = Depends(get_context)):
        if not ctx.has_role(ctx.workspace_id, at_least=role):
            raise HTTPException(403)
        return ctx
    return dep
```

Принцип: **проверка доступа в одном месте (middleware/зависимость), а не в
бизнес‑логике**. Бизнес‑логика получает уже суженный список доступных
коллекций.

### 3.4. Шеринг

- Уровни: воркспейс целиком, отдельный источник/коллекция, (позже) сохранённый
  запрос/подборка.
- `shares(resource, principal, role)`; principal = пользователь или (позже)
  «по ссылке».
- На чтении при поиске формируется **множество разрешённых коллекций** =
  собственные коллекции воркспейса + расшаренные мне. Это множество
  передаётся в multi‑store поиск (раздел 6.3).

### 3.5. Приватность как фундамент

- **Шифрование at‑rest** объектного хранилища и томов БД.
- **Право на удаление**: per‑workspace физический каталог LanceDB + записи
  Postgres + объекты S3 удаляются одной операцией (упрощено физической
  изоляцией из 2.2).
- **Аудит доступа** к источникам (`audit_log`).
- **Изоляция секретов**: ключи LLM/IdP — через секрет‑менеджер, не в коде.

## 4. Пункт 2 — Заливка файлов и фоновая доиндексация через UI

**Вероятность: ~95% (построить) / ~90% (хорошо, упирается в GPU‑очередь).**

### 4.1. Аплоад

- Прямая загрузка в S3 по **presigned URL** (большие экспорты не идут через
  API‑процесс). Для совсем больших — multipart.
- После загрузки клиент дёргает `POST /sources` → создаётся `sources` (status
  `uploaded`) и ставится job `ingest`.
- Поддержка zip/tar экспортов: распаковка в воркере в защищённую временную
  область (лимиты размера, защита от zip‑bomb — уже есть аналог
  `max_image_megapixels` для картинок).

### 4.2. Очередь задач и пайплайн

Job‑граф на источник:

```
ingest(source)            # распаковка, выбор парсера, parse → Message[] (cpu)
  └─► chunk_text          # текстовые/conversation чанки + embed (cpu)
  └─► media_caption       # VLM caption/OCR (gpu)         [опционально]
  └─► media_transcribe    # Whisper ASR (gpu)             [опционально]
  └─► document_extract    # PDF/Office/текст → чанки (cpu)
  └─► build_fts           # BM25 индекс (cpu)
  └─► graph_refresh       # инкрементальное обновление графа (cpu/gpu)
```

- **Разделение очередей `cpu`/`gpu`.** GPU — дефицитный разделяемый ресурс;
  держать отдельный пул воркеров с ограниченной конкурентностью и приоритетами,
  иначе один большой экспорт заблокирует всех.
- **Resumability уже есть**: индексация пропускает существующие `message_id`
  (`existing_message_ids`), есть `DecodePool` с таймаутами и `_HangWatchdog`.
  Это переиспользуется как механизм надёжной фоновой обработки.
- **Идемпотентность**: задачи помечают прогресс в `jobs`; повтор после сбоя
  продолжает с места.

### 4.3. Прогресс в UI

- Текущий прогресс (tqdm‑подобный счётчик в `build_index`) вынести в callback,
  пишущий `jobs.progress`.
- Доставка в UI — **WebSocket/SSE** (`/ws/jobs/<id>`), либо polling
  `GET /jobs/<id>`.

### 4.4. Дешёвое по умолчанию

Текстовая индексация (CPU) запускается сразу и быстро даёт работающий поиск;
мультимодальные шаги (GPU) — отдельные опциональные джобы (тариф/кнопка
«обогатить медиа»). Это снимает GPU с критического пути первого результата.

## 5. Пункт 3 — Любые загрузки, файлы, текст, экспорты мессенджеров

**Вероятность: ~95% (построить) / ~75% (хорошо).** Ядро надёжно; «любой
мессенджер из коробки» недостижимо — делаем как best‑effort плагины.

### 5.1. Стратегия

Опирается на абстракцию `Parser` (раздел 2.1). Два класса парсеров:

1. **Надёжное ядро** (поддерживаем и тестируем):
   - `TelegramParser` — есть (рефактор существующего).
   - `WhatsAppParser` — экспорт `_chat.txt` + медиа. Парсинг построчно с
     учётом локалей/форматов даты (хрупко по своей природе — best‑effort).
   - `GenericTextParser` — любой `.txt`/markdown → чанкуется как документ.
   - `GenericFileParser` — любой файл → извлечение текста
     (`telesearch/media/documents.py` уже умеет PDF/Office/text/CSV) →
     `split_text` → чанки. Бинарь без текста — пропускается.
2. **Плагины сообщества** (best‑effort, без гарантий): VK, Slack, Discord,
   Signal, Instagram. Реализуют тот же `Parser`, регистрируются в реестре,
   помечены как experimental.

### 5.2. «По умолчанию можно залить что угодно»

Дефолтный путь, когда формат не распознан: `GenericFileParser` /
`GenericTextParser`. Пользователь всегда получает хоть какой‑то результат
(текст извлечён и проиндексирован), даже если специализированного парсера нет.

### 5.3. Что теряется и как это честно подать

У файлов и не‑мессенджер источников нет «разговорной» структуры, на которой
держатся `conversation`‑окна и reply‑stitching (часть силы v1). Для них поиск
становится обычным документным RAG. Это нормально, но в UI стоит показывать тип
источника и не обещать «контекст переписки» там, где его нет.

### 5.4. Безопасность загрузок

- Жёсткие лимиты размера, числа файлов, глубины распаковки.
- Извлечение/декод рискованных форматов — в изолированном процессе
  (`DecodePool` уже есть) с таймаутами.
- Антивирус‑скан (ClamAV) на аплоадах как опция для прода.

### 5.5. Инструкции по экспорту

Раздел в UI с гайдами «как выгрузить данные из Telegram/WhatsApp/VK/...» (часть
уже есть в docstring `telegram_parser.py`). Чисто контентная задача, но важная
для UX.

## 6. Пункт 4 — Улучшенный поиск: фильтры, даты, комбинирование контекстов

**Вероятность: ~95% (построить) / ~90% (хорошо).** Фундамент уже сильный.

### 6.1. Структурные фильтры (prefilter)

Сейчас фильтр по модальности применяется **после** RRF, в Python
(`retriever.py`), на уже усечённом топе — это режет качество на объёме.
Переделать на **prefilter в LanceDB** (`where` внутри `hybrid_search`):

```python
def hybrid_search(self, query_text, query_vec, *, k, where: str | None):
    vec = self.table.search(query_vec).where(where, prefilter=True).limit(cand)
    fts = self.table.search(query_text, query_type="fts").where(where, prefilter=True).limit(cand)
    # ... RRF как сейчас
```

Поддерживаемые фильтры (поля уже в схеме после раздела 2.2):
`timestamp` (диапазон дат), `modality`, `source_kind`, `sender`,
`collection_id`, `lang`.

### 6.2. Модель запроса

```python
@dataclass
class SearchQuery:
    text: str
    collections: list[str]          # какие источники/контексты искать
    date_from: int | None = None
    date_to: int | None = None
    modalities: list[str] | None = None
    senders: list[str] | None = None
    source_kinds: list[str] | None = None
    k: int = 10
    rerank: bool | None = None
```

Транслируется в `where`‑выражение + скоуп коллекций.

### 6.3. Комбинирование контекстов (multi‑store fan‑out)

Из‑за per‑workspace изоляции (2.2) кросс‑воркспейс/мульти‑коллекционный поиск
делается **fan‑out + merge на уровне приложения**:

```python
def search_multi(self, q: SearchQuery, backends: list[VectorBackend]):
    pools = [b.hybrid_search(q.text, qvec, k=q.k*4, where=where) for b in backends]
    fused = rrf_merge(pools)            # та же RRF поверх нескольких источников
    return self.reranker.rerank(q.text, fused, top_k=q.k)  # единый rerank
```

- Внутри одного воркспейса коллекции живут в одной таблице → достаточно
  `where collection_id IN (...)`.
- Между воркспейсами (шеринг) → несколько backend’ов → fan‑out + общий rerank.
- Ключ: **финальный cross‑encoder rerank поверх объединённого пула** даёт
  единый, сопоставимый порядок независимо от числа источников.

### 6.4. Сохранённые подборки

«Контекст» как сохраняемый объект: набор коллекций + фильтры → именованный
пресет, которым можно искать повторно и которым можно делиться (пункт 1).

### 6.5. Масштаб

LanceDB embedded хорош для «файлы на диске». На росте (много воркспейсов,
конкурентный доступ) держать `VectorBackend` за интерфейсом и при
необходимости перейти на Qdrant/managed‑Lance/pgvector — без правок логики.

## 7. Пункт 5 — Граф знаний с кластеризацией на главном экране

**Вероятность: ~99% (построить «какой‑то граф») / ~40–55% (получить реально
полезный).** Это самая рискованная идея. Делать строго поэтапно и не обещать
сразу «граф фактов».

### 7.1. Три уровня амбиций (разная вероятность успеха)

| Уровень | Узлы | Метод | Успех |
|---|---|---|---|
| **G1** | сообщения/файлы | эмбеддинги → HDBSCAN → UMAP 2D | ~85% |
| **G2** | темы | BERTopic поверх тех же эмбеддингов; LLM именует крупные кластеры | ~55% |
| **G3** | факты/сущности + связи‑«следствия» | entity/relation extraction + GraphRAG | ~35% |

Рекомендация: **v1 = G1+G2** (почти гарантированный «вау» и реальная польза),
**G3 — отдельный R&D без обещаний в роадмапе.**

### 7.2. Пайплайн графа v1 (G1+G2)

Переиспользует уже посчитанные эмбеддинги из векторного стора — отдельная
векторизация не нужна.

```
chunks.vectors  (уже есть в LanceDB)
   │
   ├─► clustering: HDBSCAN на векторах → кластеры (узлы-темы)
   │
   ├─► layout: UMAP → 2D координаты; вес притяжения += близость по дате
   │       (узлы тем тем ближе, чем ближе факты по времени и связи)
   │
   ├─► topic labels: BERTopic (c-TF-IDF) → ключевые слова темы;
   │       LLM подписывает только крупные кластеры (экономия GPU)
   │
   ├─► node size: ∝ числу чанков/плотности темы (чем больше знаний — крупнее)
   │
   ├─► edges: косинус центроидов тем выше порога → ребро;
   │       вес = близость + пересечение участников/дат
   │
   └─► persist: graph_snapshots (узлы/рёбра/координаты) в S3 + Postgres-кэш
```

Технологии: `hdbscan`, `umap-learn`, `bertopic` (все поверх numpy‑векторов,
CPU‑friendly), LLM — только для названий крупных кластеров.

### 7.3. Производительность и рендер

- Граф **предрассчитывается в фоне** (job `graph_refresh`), не на каждый
  заход в UI; инкрементально обновляется при добавлении источников.
- Кэш снапшота с `params_hash` (фильтры/коллекции → свой снапшот).
- Рендер на **WebGL** (Cosmograph/Sigma/deck.gl) — десятки тысяч узлов.
- **Агрегация по уровням зума** (как тайлы на картах): далеко — крупные
  кластеры‑темы, ближе — подкластеры, на максимуме — отдельные сообщения.
  Иначе фронтенд умирает на тысячах точек.

### 7.4. Интерактив

Узел темы → панель с топ‑сообщениями (через обычный `Retriever` с фильтром
по теме/кластеру) и кнопкой «искать в этой теме». Граф становится навигацией
по знаниям, а не только картинкой.

### 7.5. G3 (R&D, опционально, позже)

Если двигаться к «узлам‑фактам»: GraphRAG‑подход (LLM извлекает
сущности/отношения → агрегирует в граф сообществ). Честные оговорки: дорого по
токенам/GPU, нестабильно на шумной живой переписке (сленг, сарказм,
мультиязычность), требует агрессивной чистки от «волосатого шара». Только как
эксперимент с измеримой оценкой пользы, не как обещание.

---

## 8. Сквозные нефункциональные требования

- **GPU‑экономика — главное бутылочное горлышко, не алгоритмы.** Каждый новый
  пользователь с медиа = часы GPU. Решения: отдельная gpu‑очередь с
  приоритетами/квотами; текст бесплатно/сразу, медиа по запросу/в тарифе;
  батчинг VLM/ASR; автоскейл gpu‑воркеров.
- **Приватность/комплаенс** (см. 3.5) — обязательны, не опциональны.
- **Наблюдаемость**: трассировка job’ов сквозь очередь (OTel), метрики глубины
  очередей и времени индексации, алерты.
- **Стоимость хранения**: дедуп медиа по хэшу; lifecycle‑политики на S3.
- **Лимиты/квоты** на размер загрузок, число источников, объём индекса.

## 10. Сводка вероятностей технического успеха

| # | Направление | Построить | Хороший результат | Главный риск |
|---|---|---|---|---|
| 1 | Auth / юзеры / шеринг / изоляция | ~99% | ~95% | приватность, не код |
| 2 | Загрузки + фоновая индексация | ~95% | ~90% | GPU‑очередь |
| 3 | Любые загрузки + парсеры | ~95% | ~75% | «любой мессенджер» недостижим |
| 4 | Фильтры + комбинирование контекстов | ~95% | ~90% | масштаб векторного стора |
| 5 | Граф знаний | ~99% | 40–55% | G1≈85%, G2≈55%, G3≈35% |

**Главный вывод.** Это не «расширение фич», а смена класса проекта: из
локальной утилиты в SaaS. Технически осуществимо целиком; ни один пункт не
требует недоступной технологии. Наибольшая работа — не в пяти пунктах, а в
фундаменте под ними (Фазы 0–1). Два настоящих ограничителя — **GPU‑экономика**
(определит архитектуру и тариф сильнее любого кода) и **приватность**
(диктует изоляцию на уровне БД и делает её приоритетнее графа). Граф —
единственное направление с реальной неопределённостью результата: строить
поэтапно (G1→G2→G3) и не обещать «граф фактов» до R&D‑подтверждения.
